# Remediation Prompt — Android Real-Data and Primary-Flow Completion

_Closes gap:_ G5 · android-real-data-and-primary-flow-completion

## Context

The Android app has strong Compose/Hilt/Room/Retrofit scaffolding and many tests, but primary seller screens fall back to hard-coded orders, dishes, and reviews; `SellerViewModel` does not load a business; settings and uploads contain no-op actions; saved cards are mock values; and offline orders lose line items. This creates a polished demo that can misrepresent disconnected or failed production behavior.

## What to build

Make repositories the only runtime data authority, define explicit state machines for seller/customer flows, preserve complete offline order mutations with idempotent sync, finish settings/media/payment interactions, and delete duplicate screen implementations after navigation and tests point to canonical screens. Consume the canonical API generated by G4 rather than adding new handwritten paths.

## UI design constraints

- Existing product style is authoritative; the audited Material 3 theme and current navigation are the design evidence for this completion work.
- Preserve Material 3 colors/typography from `ui/theme/`; this is completion, not redesign.
- Use one state model for default, loading, empty, stale/offline, error, disabled, and success.
- Dashboard behavior must define each KPI, filter, chart, table, tooltip, and legend, including loading, empty, and error behavior without fabricated values.
- Maintain 48dp targets, TalkBack labels, focus order, RTL layout, dynamic font scaling, and light/dark previews.
- Add screenshot baselines for seller dashboard, empty menu, offline order queue, and payment failure after the real-data changes.

## What NOT to do

- Do not use sample content as an empty-state fallback or hide repository errors.
- Do not sync partial order aggregates that cannot reconstruct line items and totals.
- Do not create another endpoint registry, repository, Cart screen, or Menu screen.
